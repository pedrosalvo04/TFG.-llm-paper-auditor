# 02 — Functional Frontend Specifications

This document defines the functional requirements and behavior of the Streamlit frontend for the NeurIPS 2026 Reproducibility Auditor.

## 1. Application Bootstrap & Initialization

**Entry Point**: `frontend/app.py`
**Alternate Root**: `app.py` (project root)

The application initialization sequence is as follows:
1.  **Environment Setup**: `app.py` (root) sets critical environment variables (`TRANSFORMERS_VERBOSITY`, `TOKENIZERS_PARALLELISM`, etc.) before importing `frontend.app`.
2.  **Page Configuration**: `st.set_page_config` defines the title, layout ("wide"), and ACM logo icon.
3.  **Styling**: `apply_custom_styles()` (from `frontend/styles/custom_css.py`) injects the global CSS theme.
4.  **State Initialization**: `initialize_session_state()` ensures all required keys exist in `st.session_state`.

## 2. Session State Management

**Module**: `frontend/utils/session_state.py`

| Key | Type | Initial Value | Purpose |
|---|---|---|---|
| `resultado` | `dict` | `None` | Stores the final audit output from `PaperAuditor.audit()`. |
| `auditor` | `PaperAuditor` | `PaperAuditor()` | Singleton instance of the backend auditor service. |
| `chatbot` | `PaperChatbot` | `PaperChatbot()` | Singleton instance of the chatbot service. |
| `sota_analyzer` | `SotaAnalyzer` | `SotaAnalyzer()` | Singleton instance of the SOTA analyzer service. |
| `messages` | `list` | `[]` | Chat history for the PaperChatbot. |
| `md_text` | `str` | `None` | The Markdown text extracted from the uploaded PDF. |
| `archivo_actual` | `str` | `None` | Name of the file currently processed (for deduplication). |
| `file_hash` | `str` | `None` | MD5 hash of the file content (for deduplication). |

## 3. UI Component: File Uploader

**Module**: `frontend/components/file_uploader.py`

**Function**: `process_uploaded_file(uploaded_file)`

| Logic Step | Description |
|---|---|
| **Trigger** | User selects a file in `st.file_uploader`. |
| **Deduplication** | Checks if `uploaded_file.name == st.session_state.archivo_actual` and if the content hash matches `st.session_state.file_hash`. If both match, the audit is skipped. |
| **PDF Processing** | 1. Saves file to `./temp/`.<br>2. Calls `convert_pdf_to_markdown(temp_path)`.<br>3. Stores result in `st.session_state.md_text`. |
| **Audit Execution** | Calls `st.session_state.auditor.audit(md_text, status_callback)`. |
| **Result Storage** | Writes the returned dict to `st.session_state.resultado`. |
| **Cleanup** | Deletes the temporary PDF file. |

## 4. UI Component: Audit Results (Compliance Table)

**Module**: `frontend/components/audit_results.py`

**Function**: `render_audit_results(resultado, uploaded_file)`

Renders the primary compliance dashboard.
-   **Header**: Displays "Ficha Técnica" with paper type and metadata.
-   **Metrics Row**: Displays execution time, paper length, and detected red flags.
-   **Compliance Table**: An HTML table generated by `_build_table_html(resultado)`.
    -   **Columns**: Requirement, Verdict (Yes/No/NA), Evidence/Justification.
    -   **Styling**: Rows are colored based on verdict (Green for Yes, Red for No, Gray for NA).
-   **Risk Alerts**: Displays a warning box if `get_checklist_health()` returns a "risk" status.

## 5. UI Component: Gauge Chart

**Module**: `frontend/components/gauge_chart.py`

**Function**: `create_gauge_chart(score)`

Renders a Plotly-based gauge indicator showing the overall reproducibility score (0-100%).
-   **Tiers**: Categorizes the score into "Strong Accept", "Accept", "Borderline", "Weak Reject", "Reject", or "Strong Reject".
-   **Colors**: Dynamic bar color corresponding to the quality tier.

## 6. UI Component: SOTA Analysis Section

**Module**: `frontend/components/sota_section.py`

**Function**: `render_sota_analysis(md_text)`

-   **Trigger**: User navigates to the "SOTA Analysis" tab.
-   **Logic**: Calls `st.session_state.sota_analyzer.analyze_sota(md_text)`.
-   **Display**:
    -   **Thematic Summary**: Identified subtopics and technical areas.
    -   **Recent Papers**: A table of top 10 relevant papers from Semantic Scholar with citation counts.
    -   **Gap Analysis**: Identified weaknesses in the paper's literature review.

## 7. UI Component: Paper Chatbot

**Module**: `frontend/components/chatbot.py`

**Function**: `render_chatbot(md_text)`

-   **Interface**: A standard chat interface using `st.chat_message` and `st.chat_input`.
-   **Logic**:
    1.  User submits a question.
    2.  Appends message to `st.session_state.messages`.
    3.  Calls `st.session_state.chatbot.preguntar(md_text, question, history)`.
    4.  Displays response using a "Senior Area Chair" persona.

## 8. Frontend Utilities: Scoring & Reports

**Module**: `frontend/utils/scoring.py`
-   **`get_checklist_health(evaluation)`**: Analyzes the 16 checklist items to identify "unjustified No" or "Yes without evidence" items. Returns a status dict.

**Module**: `frontend/components/audit_results.py` (Report Generation)
-   **`generate_report(resultado, uploaded_file, health_data)`**: Combines the audit data and health analysis into a downloadable PDF report (using `reportlab`) or Markdown file.
